# UI使用制度（设计资产 → 开发落地）

版本：v1.0  
适用范围：本项目所有 UI 相关交付（设计源文件、页面参考、可复用导出资源、规范文档）  
生效时间：即日起  

## 1. 目标与原则

### 1.1 目标

- 让设计资产可追溯：任何页面与资源都能回到“设计真源”和对应迭代说明
- 让开发可复现：开发不依赖临时沟通即可按规范实现一致 UI
- 让迭代可控：变更有版本、有评审、有记录，可回滚、可对比

### 1.2 核心原则（必须遵守）

- 设计真源唯一：Figma 源文件是唯一真源；截图只做参考留档，不作为最终交付依据
- 导出物制度化：只有按规范导出的资源允许进入工程引用
- 变更要可见：任何影响 UI 的变更必须有版本号与变更记录
- 先骨架后细节：先统一布局/组件/状态规则，再补充具体页面细节

## 4. 命名规范（必须用于新文件）

### 4.1 页面参考截图（Reference）

命名格式：

`<端>-<模块>-<页面>-<状态>-v<版本>.<png|jpg>`

示例：
- pc-auth-login-default-v1.png
- pc-dashboard-home-default-v1.png
- app-auth-login-default-v1.png

字段约定：
- 端：pc / app
- 模块：dashboard / order / customer / message / settings 等
- 状态：default / empty / loading / error / modal / drawer / readonly 等
- 版本：从 1 开始递增（仅用于参考文件的自洽管理）

### 4.2 导出资源（Exports）

命名格式：

`<类型>/<业务域>/<名称>@<倍率>.<格式>`

示例：
- icons/common/chevron-right@1x.svg
- icons/common/chevron-right@2x.svg
- logos/brand/store-mark.svg
- images/order/empty-state@1x.png

约定：
- Icon：优先 SVG；如必须 PNG，需提供 @1x/@2x 或指定尺寸
- 插画/图片：固定尺寸 PNG；若需要可缩放则用 SVG

## 5. 导出制度（谁导出、导出什么、如何进入工程）

### 5.1 允许进入工程的资源类型

- icons：通用图标、业务图标
- logos：品牌/门店标识
- illustrations：引导插画/空状态插画
- images：少量固定用途图片（如空状态背景、引导图）

### 5.2 禁止进入工程的文件

- 整页截图（Reference）
- 未经规范命名与版本记录的导出物
- 无法回溯来源的图片（不知道来自哪个页面/组件/版本）

### 5.3 导出流程（最小可执行）

1) 设计侧从真源导出资源，放入 UI设计/20_exports/ 对应子目录  
2) 在 UI设计/CHANGELOG.md 增加一条记录（导出范围、原因、版本号、影响面）  
3) 开发侧只能从 20_exports 引用资源；不得从 10_reference 或旧截图目录引用  

## 6. 版本与变更管理

### 6.1 版本规则（建议）

- vX.Y
  - X：破坏性变化（影响现有实现或语义变化，如状态规则/组件结构/布局体系变化）
  - Y：兼容性变化（新增组件、新增状态、修复样式细节）

### 6.2 变更日志（CHANGELOG.md）最小格式

- 日期
- 版本号
- 变更摘要（1-3条）
- 影响范围（哪些页面/组件/资源）
- 验收说明（如何验证）

### 6.3 禁止覆盖规则（防止“引用被悄悄替换”）

- 一旦导出资源被工程引用，不允许用同名文件直接覆盖不同内容
- 需要更新时：
  - 要么新增版本文件名
  - 要么提升目录层级的版本并同步更新引用点

## 7. 评审制度（设计评审 + UI验收）

### 7.1 必须评审的变更

- 新增页面/弹窗/抽屉
- 新增或修改核心组件（表格、筛选器、状态徽标、Drawer/Modal、时间线）
- 状态颜色/文案/规则变更
- 任意导出资源变更（icons/logos/illustrations/images）

### 7.2 评审材料最小集（MVP）

对照基线：
- 以“前端页面清单与Figma线框说明稿（MVP）”为页面范围基线
- 以“后台页面原型结构图（MVP）”为页面结构与关键路径基线

每个页面至少提供：
- default
- empty
- loading
- error

对工单详情页额外提供（建议）：
- waiting_approval
- waiting_pickup
- completed（只读态）

### 7.3 评审结论落档

- 评审结论必须写入 CHANGELOG（通过/修改后通过/拒绝 + 结论摘要）

## 8. 开发落地流程（无前端工程也可执行）

### 8.1 开发前置输出（必须先有）

- ui-tokens-core.md（颜色/字号/间距/圆角/阴影等 Token，先用表格文本也可）
- components-admin.md（后台组件清单：名称、用途、状态、交互要点）

### 8.2 组件落地顺序（推荐）

1) Layout：顶部栏/侧边栏/内容区
2) 基础组件：按钮、输入、选择、标签、徽标、提示
3) 表格与筛选器：列表页核心
4) Drawer/Modal：新建与高风险操作的统一交互
5) 业务组件：工单状态切换、消息卡片、时间线

### 8.3 验收口径（开发自检）

- 不依赖截图逐像素对齐，而依赖：
  - Token 是否一致（颜色/字号/间距）
  - 组件状态是否齐全
  - 页面结构是否符合 MVP 文档
  - 高风险操作是否有二次确认与明确提示
